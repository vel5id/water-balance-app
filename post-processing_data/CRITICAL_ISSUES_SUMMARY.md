# 🚨 КРИТИЧЕСКИЙ АНАЛИЗ: Почему Prophet показал отвратительные результаты

## 📊 Обнаруженные проблемы

### **1. КРИТИЧЕСКАЯ ОШИБКА: Неправильное количество origins**

```python
# ❌ ПРОБЛЕМА (исходный код):
step = max(1, (end_i - start_i) // 200)  # Ограничение: максимум 200 origins

# ✅ ПРАВИЛЬНО (исправлено):
step = max(1, (end_i - start_i) // 800)  # Как в SARIMA: ~800 origins
```

**Результат:**
- Prophet: **212 origins** (первые запуски)
- SARIMA: **1474 origins** (справочно из SARIMA backtest)
- **Несправедливое сравнение!** Prophet тестировался на **в 7 раз меньшем** наборе данных

**Почему это проблема:**
- Меньше origins = меньше статистическая значимость
- Возможны случайные выбросы, не представляющие истинную производительность
- Метрики (R², RMSE) могут быть искажены из-за малой выборки

---

### **2. КАТАСТРОФИЧЕСКАЯ ОШИБКА: Индексация прогнозов**

```python
# ❌ БЫЛО (НЕПРАВИЛЬНО):
predictions = forecast.tail(future_days).set_index('ds')['yhat']

for h in horizons:
    target_date = dates[i] + pd.Timedelta(days=h)
    pred = predictions.loc[target_date]  # ❌ target_date НЕ В predictions.index!
```

**Проблема:**
- `forecast` содержит прогноз для **ВСЕГО периода** (история + будущее)
- `forecast.tail(future_days)` возвращает **последние N строк**, не связанные с `origin_date`
- **Даты не совпадают!** `target_date` не находится в `predictions.index`
- Результат: **многие прогнозы были NaN**, что приводило к отрицательным R²

**Пример проблемы:**
```
История: 2020-01-01 до 2020-04-10 (100 дней)
future_days = 365
forecast: 2020-01-01 до 2021-04-10 (465 строк)
forecast.tail(365): 2020-04-11 до 2021-04-10

origin_date = 2020-04-10
target_date (h=30) = 2020-05-10 ✅ Есть в прогнозе

НО! Следующий origin:
origin_date = 2020-04-20
target_date (h=30) = 2020-05-20 ❌ НЕТ в прогнозе (прогноз был до 2021-04-10)
```

**Исправление:**
```python
# ✅ ПРАВИЛЬНО:
forecast_df = forecast[['ds', 'yhat']].set_index('ds')  # Весь прогноз, не tail()

for h in horizons:
    target_date = origin_date + pd.Timedelta(days=h)
    pred = forecast_df.loc[target_date, 'yhat']  # Теперь работает!
```

### **3. ОШИБКА: Неправильная логика скользящего окна**

```python
# ❌ БЫЛО (НЕПРАВИЛЬНО):
def _fit_one_origin(i: int):
    hist = vol.iloc[: i + 1]  # i - индекс из range(), не индекс Series!
```

**Проблема:**
- `i` — это индекс в `range(start_i, end_i, step)`, например `i = 150`
- `vol` имеет **DatetimeIndex**, не integer index
- `vol.iloc[:151]` возьмёт **первые 151 строки**, независимо от `dates[i]`
- Результат: **история не соответствует origin_date**

**Исправление:**
```python
# ✅ ПРАВИЛЬНО:
def _fit_one_origin(i: int):
    origin_date = dates[i]  # Сначала получаем дату
    hist = vol.loc[: origin_date]  # Используем .loc с датой
```

## 🎯 ВЫВОДЫ

### **1. Prophet ПРОВАЛИЛСЯ на коротких горизонтах**
- **1-30 дней:** SARIMA в **10-20 раз лучше**
- Prophet не может уловить краткосрочные флуктуации
- Отрицательные R² указывают на **систематическую ошибку**

### **2. Prophet показывает потенциал на 365 дней**
- **RMSE 94.69 vs SARIMA 224.20** — в 2.4 раза лучше!
- Но R² = -3.94 всё ещё отрицательный
- **Причина:** Нужны внешние регрессоры (температура, осадки, приток)

### **3. Гиперпараметры критичны**
- `changepoint_prior_scale`:
  - 0.05: Слишком жесткий (R² = -68)
  - 0.5: **КАТАСТРОФА** (R² = -514!)
  - 0.01: Оптимально (R² = -3.94)
- **Вывод:** Нужна тонкая настройка для каждого датасета

### **4. Multiplicative seasonality важна**
- Для водохранилищ сезонность **растёт с объёмом**
- `seasonality_mode='multiplicative'` даёт лучшие результаты
- `additive` приводит к переобучению

### **5. Нужно больше данных для обучения**
- Prophet требует **минимум 1-2 года истории** для качественных прогнозов
- С 90-дневной историей модель не может уловить годовую сезонность
- **Рекомендация:** Увеличить `start_i` до 365 дней
